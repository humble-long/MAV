## 3.1 总体框架

### 3.1.1 设计动机

工程设计场景下的领域问答与方案推荐有两点显著区别于通用百科问答。其一，问题往往伴随定量约束："设计一架续航 30 min、载重 10 g 的扑翼机需要多大起飞重量？"——这要求答案不仅在语义上自洽，还要在物理量纲上严格成立。其二，问题的可信度依赖于推理过程的可追溯性，而非单一答案。然而，纯大语言模型受限于训练分布，其给出的数值常缺乏可验证依据；纯知识图谱检索虽然事实可控，但难以将知识图谱中的离散三元组转化为面向用户问题的连贯解释。为此，本文针对仿生扑翼飞行器领域定制了一套面向尺度律的物理工具集（详见 3.2 节），并在 ReAct 范式下与第 2 节的双层知识图谱协同：让 LLM 充当推理控制器，在双层本体上沿语义关系（含 4 类 MIMICS 通道）展开多跳检索，并在需要数值验证的步骤主动调用封装好的物理工具，形成"知识检索 → 物理校验 → 答案合成"的可解释推理链。本节工作的原创性主要体现在两处：（i）针对仿生扑翼飞行器尺度律推导出的 4 个领域专用物理工具及其量纲一致的接口设计；（ii）将这些工具与双层本体路径检索协同形成的可解释问答与精排范式——而非 ReAct 或 Function Calling 协议本身。

### 3.1.2 推理范式

整体推理流程采用 ReAct 范式<sup>[Yao2023]</sup>，并借鉴 ToG<sup>[Sun2023]</sup> 的"LLM 在图上推理"思想与 RoG<sup>[Luo2024]</sup> 的关系路径规划思想。在每一轮迭代中，LLM 根据当前已观测到的事实自主选择以下三类动作之一：（i）发起一次工具调用，可以是 KG 检索工具（如查询某机型的属性、查询 MIMICS 仿生映射）也可以是物理工具（如调用 Hassanalian 重量模型）；（ii）综合现有证据给出最终答案；（iii）声明当前证据不足以作答。工具执行结果以"观察"的形式回写到对话上下文，下一轮 LLM 再据此决定下一步动作，直至给出最终答案或达到最大迭代次数（默认 D = 6）。该范式与 ToG 的关键差异在于：ToG 在图上做关系层面的束搜索，而本文将"图遍历 + 数值校验"统一为可调用工具集——LLM 看到的不是图的结构 API，而是 7 个具语义的工具入口（图 3）。

## 3.2 物理工具库设计

### 3.2.1 4 个尺度律物理工具

为了让 LLM 在推理过程中执行物理验证，本文从仿生扑翼飞行器领域的经典尺度律与气动判别准则中提炼出 4 个核心工具，每个工具以确定性函数形式实现，输入与输出均带量纲标注，以保证与第 2.1.3 节量纲感知属性的一致性。4 个工具的规格如表 2 所示。

**表 2** BioBridge-GraphRAG 物理工具集

| 工具名 | 功能 | 输入（带单位） | 输出（带单位） | 公式来源 |
|---|---|---|---|---|
| `hassanalian_weight` | 由任务约束反推起飞重量与子系统重量分配 | endurance (min); payload (g); avionics (g); battery_density (Wh/kg) | total_weight (g); battery (g); structure (g); 可行性标志 | Hassanalian 2017<sup>[Hassanalian2017]</sup> |
| `shyy_scaling_law` | 由起飞重量预测翼展、翼面积、翼载荷、扑频 | weight (g) | wingspan (mm); wing_area (cm²); wing_loading (N/m²); freq (Hz); v_min_power (m/s); aspect_ratio (—) | Shyy 2013<sup>[Shyy2013]</sup> + Pennycuick 1996<sup>[Pennycuick1996]</sup> |
| `strouhal_check` | 计算 Strouhal 数 St = fA/U 并判别是否在最优区间 | freq (Hz); amplitude (m); speed (m/s); 可选 wingspan (mm) | St (—); 是否在 0.2–0.4 区间 | Triantafyllou 1991<sup>[Triantafyllou1991]</sup> |
| `reynolds_check` | 计算雷诺数 Re = Uc/ν 并判别流态 | chord (mm); speed (m/s); altitude (m); temperature (°C) | Re (—); 流态分类（昆虫前缘涡 / 中小昆虫 / 过渡 / 鸟类） | Shyy 2013<sup>[Shyy2013]</sup> §1.3 |

除上述 4 个物理工具外，工具集还包含 3 个 KG 检索工具（`search_fwmav` / `search_organism` / `query_mimics_path`）用于按属性筛选样机、查询生物原型参数与查询双层本体上的 MIMICS 映射关系，以及 1 个张量分解粗筛工具 `tensor_recall`（详见第 4 节）。8 个工具的统一入口构成了 LLM 在推理过程中的全部行动空间。

### 3.2.2 基于 Function Calling 的工具调用接口

每个工具按照 OpenAI Function Calling 协议<sup>[OpenAI2023]</sup>给出标准化的 JSON Schema 描述，包括：函数名、自然语言功能说明、参数列表（含类型、是否必填、单位、默认值与枚举取值）。LLM 在每一轮推理时同时接收用户当前问题、已有观察以及全部工具的 JSON Schema，并以结构化的工具调用对象作为输出。该协议被主流 LLM 服务（OpenAI GPT、DeepSeek、Anthropic Claude 等）广泛兼容，使得本文方法能够在不同 LLM 后端之间无缝迁移。所有工具均以确定性数值函数实现并集中注册于一个调度入口，避免了将物理公式硬编码到 LLM 提示中带来的可维护性问题；当尺度律或经验系数发生更新时，只需修改对应的工具实现，推理流程本身不需调整。

## 3.3 路径推理算法

### 3.3.1 算法描述

完整的路径推理流程如算法 1 所示。算法以用户查询 $Q$ 为输入，以双层知识图谱 $\mathcal{G}$、LLM 控制器 $\mathcal{M}$ 与工具集 $\mathcal{T}$ 为推理资源，输出最终答案 $A$ 与完整的推理轨迹 $\mathcal{P}$（包含每一步的思考、动作、观察）。

```
算法 1 BioBridge 工具增强路径推理（ReAct 范式）
─────────────────────────────────────────────────────────────
输入：用户查询 Q；知识图谱 G；LLM 控制器 M；工具集 T；
      最大迭代步数 D
输出：最终答案 A；推理轨迹 P

 1: messages ← [系统提示, 用户查询 Q]
 2: P ← []
 3: for d = 1 to D do
 4:     response ← M.chat_with_tools(messages, T.specs())
 5:     if response.tool_calls 不为空 then
 6:         for each (tool, args) ∈ response.tool_calls do
 7:             result ← T.call(tool, args)        ▷ 工具执行
 8:             P.append({step: action,    tool, args})
 9:             P.append({step: observation, tool, result})
10:             messages.append(工具调用记录, 观察结果)
11:         end for
12:     else if response.content 不为空 then        ▷ LLM 给出答案
13:         A ← response.content
14:         P.append({step: final_answer, content: A})
15:         return A, P
16:     end if
17: end for
18: return "达到最大迭代步数", P
─────────────────────────────────────────────────────────────
```

算法的核心特点是**控制器与工具解耦**——LLM 不直接访问 KG 或公式，而是通过工具集间接获取，所有数值计算都被显式记录在轨迹 $\mathcal{P}$ 中，便于后续审计与可解释性分析。

### 3.3.2 关键设计点

**工具触发的隐式策略**。算法不显式判别"何时该调工具"，而是将工具的语义说明（包括典型适用场景）写入 JSON Schema 的 `description` 字段，由 LLM 在推理时自行决策。例如 `hassanalian_weight` 的描述中明确列出"用户问『做一架续航 X 分钟、载重 Y g 的扑翼机需要多重』时调用"，使得 LLM 在遇到此类约束时能够主动触发该工具。

**单路径 ReAct vs. 束搜索**。当前实现采用单路径 ReAct，每轮只展开一条推理链；算法可平滑扩展为 ToG 风格的束搜索（每轮保留 $k$ 条候选路径），代价是 LLM 调用次数从 $O(D)$ 增长到 $O(kD)$。在第 5 节定义的 6 类典型查询上的初步实验表明，单路径 ReAct 在 $D = 6$ 内即可收敛（6 个代表性 demo 的平均迭代数为 4.0、最大 6，平均工具调用次数为 6.8、最大 10），束搜索带来的额外开销与边际收益尚不显著。

**工具失败的容错**。工具调用可能因参数越界（如载重为负数）或物理不可行（如续航需求与电池密度不匹配）而失败。失败信息以 JSON 形式回写到上下文，LLM 据此自行修正下一步动作（如调整参数重试、换用其他工具或直接回答"不可行"）。这一容错机制使得推理链对参数错误具有自愈能力，无需在控制层加入复杂的重试逻辑。

## 3.4 应用模式

本文的工具增强路径推理在 BioBridge-GraphRAG 中承担两类工作（图 4）：知识查询模式与推荐精排模式。

**知识查询模式**面向单点或对比型问题。例如查询"鸽子和隼的扑频差异在哪？"时，LLM 依次调用两次 `search_organism` 获取两类生物的扑频区间，比较后给出对比表与文献溯源；查询"扑频 18 Hz、翼展 280 mm、巡航速度 6 m/s 的扑翼机 St 数是否合理？"时，LLM 调用 `strouhal_check` 完成数值校验并返回是否在 0.2–0.4 最优区间；查询"找一架以形态相似为主导仿生于鸽子的样机"时，LLM 通过 `query_mimics_path` 工具按 `dominant_type=morphology` 过滤 MIMICS 边，仅返回形态主导（而非尺度主导）的候选样机，体现了第 2 节 4 类相似度通道的细分价值。该模式下推理深度通常较浅（$D \leq 3$），工具调用次数为 0–3 次。

**推荐精排模式**面向方案设计型问题，与第 4 节的张量分解粗筛协同形成"粗筛—精排"两阶段范式。具体而言，给定任务约束（如"重量 30 g、续航 15 min、需悬停的微型扑翼机"），张量分解粗筛先返回 Top-$K$ 候选样机（默认 $K = 10$）；本节路径推理对每个候选依次调用 `hassanalian_weight` 复核任务约束下的可行重量、调用 `shyy_scaling_law` 比较其参数与尺度律预测的偏离度、调用 `query_mimics_path` 检查其仿生主导维度是否与任务匹配，最终输出 Top-3 推荐及附带物理校验报告与推理路径。物理校验报告以 JSON 结构记录"工具名—输入参数—输出数值—物理可行性结论"四元组，并在前端按候选样机渲染为可读的对比表（最小示例见附录 A）。该模式下推理深度较深（$D \approx 4$–6，由 demo 实验所得；详见 3.3.2），是论文实验部分（第 5 节）方案推荐任务的核心评测对象。
